# 项目执行文件包目录标准
## 1. 目录定位
当 `00-项目包` 已确认可用，并准备进入真实需求执行时，应由总控派生“项目执行文件包”。
它解决的是：
- 当前项目这次怎么落
- 模块级产物放哪里
- 完整提交包放哪里
- 正式产物和过程产物如何分层
## 2. 两种提交方式
产品经理有两种标准提交方式：
1. **分模块提交**
   - 适合日常推进、分模块评审、局部迭代
   - 每个模块有自己的执行包
2. **完整提交**
   - 适合阶段汇报、里程碑评审、一次性交付
   - 把多个模块的正式产物汇总成一个完整视图
如项目需要，同时保留这两种方式。
## 3. 派生规则
- 项目包未确认完成前，禁止派生下游执行目录。
- 若当前项目是历史项目，进入目录派生前必须先完成历史项目接入分型，并显式区分：
  - `历史项目首次接入`
  - `已接入历史项目但项目级入口不完整`
  - `已接入历史项目但派生骨架不完整`
- 派生前，必须先明确本次采用哪种交付模式：
  - `分模块推进`
  - `完整提交`
  - `模块推进 + 完整提交`
- 派生只负责创建目录骨架和说明文件，不自动生成模块正文。
- 若两个及以上 Agent 的预期写入目标重叠，例如同一正式文件、同一正式章节、同一页面编号集合、同一审查记录或同一提交状态链路，必须由总控拆成严格串行依赖链，禁止并行派发。
### 3A. 两段式派生固定顺序

- `第一段：项目级真源派生`
- `第二段：下游执行目录派生`
- 第一段未完成前，不得启动第二段。
- 第二段完成前，必须补做：
  - `同层编号唯一性`
  - `应有骨架完整性`
  - `事实源与交付视图边界`
## 4. 标准目录结构
推荐在真实项目目录下使用以下结构：
```text
项目目录/
  AGENTS.md
  00-项目包/
    01-项目商业说明.md
    02-用户画像与权限架构.md
    03-系统拓扑与核心名词表.md
    05-项目级原型与高保真标准.md
    06-产品级共享资产底座.md
    02-工程交接/
      README.md
    04-过程文件/
      00-沟通记录与结论回写.md
      01-总控复核清单.md
  01-模块执行包/
    01-模块A/
      00-说明.md
      01-PRD/
      02-工程交接/
      02-原型/
        手机端/
        PC端/
        管理后台/
      03-评审提交包/
      04-过程文件/
    02-模块B/
  80-完整提交包/
    00-提交状态.json
    00-提交说明.md
    01-合并PRD/
      00-PRD索引.md
    02-原型索引/
      00-原型入口.md
    03-合并评审记录/
      00-评审索引.md
      30-结果-AI深审记录.md
  docs/
    00-说明.md
    01-跨模块补充资料/
      README.md
      local/
      published/
  scripts/
    sync_submission_status.py
    check_submission_consistency.py
    check_engineering_handoff_readiness.py
    check_final_delivery_gate.py
    check_publish_boundary.py
```
历史项目如果已经保留 `90-工具包/`，可以继续兼容存在；新接入项目不要求继续创建这类目录。
兼容说明：
- 当前方法论仍保留既有 `02-原型/` 路径合同，同时新增 `02-工程交接/` 作为 Gate B landing zone。
- 这意味着模块目录中会并存两个以 `02-` 开头的目录；这是兼容旧项目路径的显式设计，不是笔误。
- 后续若要统一重编号，必须连同规则、模板、脚本与真实项目迁移方案一起改，不能在单个模板里静默改名。
## 5. 模块执行包说明
每个模块目录至少包含：
- `00-说明.md`
  - 说明当前模块范围、状态、对应项目包版本
- `01-PRD/`
  - 放当前模块正式 `PRD`
  - 固定放 `02-模块流程图与人类确认.md`，作为每模块必备的正式下游文件
  - `02-模块流程图与人类确认.md` 承接页面流转、点击流、规则卡核对与人类确认，不重写 `PRD` 正文
  - 放问题响应记录、业务取舍说明
- `02-原型/`
  - 按终端拆分
- `03-评审提交包/`
  - 放 `AI深审记录`
  - 放跨角色预审讨论清单
  - 放评审纪要等正式提交材料
- `02-工程交接/`
  - 放 `FE / BE / QA / DBA / Ops/SRE` 的 Gate B 技术真源
  - 放 trigger-required 的状态图、策略说明、生命周期说明等 supporting docs
  - 不能用阅读层摘要或 `80-完整提交包/` 展示层代替
- `04-过程文件/`
  - 固定包含 `00-沟通记录与结论回写.md`
  - 进入四审前和四审收口后，默认新增或更新 `01-总控复核清单.md`
  - 放草稿、中间审查结果、子 agent 输出、临时文件
  - 任何已确认的新结论，先保留完整问答，再把确认结论回写到正式文件
  - 所有执行与审查 Agent 在交还控制权前，必须先输出固定格式的 `工单回执`；总控验收通过后，才能继续判断下一跳
## 6. 过程文件落盘判定规则
任何中间材料在落盘前，必须先判断它属于哪一层：
| 材料类型 | 正确位置 | 说明 |
|---|---|---|
| 单一模块的 spec、plan、草稿、子 agent 输出、临时分析 | `01-模块执行包/<模块名>/04-过程文件/` | 默认归属到对应模块，不能偷懒放根目录 |
| 单一模块的四审前后总控复核清单 | `01-模块执行包/<模块名>/04-过程文件/01-总控复核清单.md` | 强制项展开、二次对照、未覆盖项清零都在这里留痕 |
| 跨多个模块、但仍服务真实项目推进的 spec、plan、阶段纪要、全局分析 | `00-项目包/04-过程文件/` | 项目级过程材料默认放这里 |
| 跨模块的四审前后总控复核清单 | `00-项目包/04-过程文件/01-总控复核清单.md` | 无法判断归属时也先落这里，不放根 `docs/` |
| 非事实源、跨模块、且已经明确批准保留的补充资料 | `docs/01-跨模块补充资料/` | 必须满足白名单规则，不得替代事实源 |
明确禁止把以下路径当默认落盘位置：
- `docs/superpowers/`
- `docs/plans/`
- `docs/specs/`
- 任何未经批准的根 `docs/` 自建目录
如果无法立刻判断归属层级，默认先进入 `00-项目包/04-过程文件/`，而不是落到根 `docs/`。
执行类 Agent 默认不读取 `04-过程文件/`，避免把过程痕迹误当成事实源或主上下文。
只有命中明确职责时，才允许显式读取过程文件：
- `总控 / 门禁 / 四审 / 收口` 可以显式读取 `01-总控复核清单.md`
- `记录同步 / 冲突追因 / 复盘 / 审计` 可以显式读取 `00-沟通记录与结论回写.md`
## 7. 根 `docs/` 白名单规则
根 `docs/` 不禁用，但只允许承接“非事实源、跨模块、明确批准”的补充资料。
因此真实项目仓库必须同时具备：
- `docs/00-说明.md`
- `docs/01-跨模块补充资料/README.md`
放进 `docs/01-跨模块补充资料/` 的文档，头部至少要写清：
- 这不是事实源
- 为什么必须跨模块保留
- 谁批准放这里
- 对应的正式事实源文件有哪些
同时必须区分两层边界：
- `docs/01-跨模块补充资料/local/`：只允许本地保留，不进入远端 git
- `docs/01-跨模块补充资料/published/`：只有明确批准后，才允许进入远端 git
## 8. 完整提交包说明
`80-完整提交包/` 只负责提供完整提交视图，不是事实源。
- 项目级与模块级工程交接 README、`80-完整提交包/` 展示层都是消费镜像，不是另一套运行时真源。
- `engineering_handoff_required=false` 时，模块级绿灯只看基础四审。
- `engineering_handoff_required=true` 时，模块级绿灯 = 基础四审 + `24-审查-跨角色预审` 完成 + Gate B 绿灯。
- `24-审查-跨角色预审` 只是 Gate B 前置条件，不是第二条独立绿灯通道。
- Gate B 写回只允许 `engineering_handoff_required / engineering_handoff_ready`。
- `ready_for_human_review` 与 `engineering_handoff_ready` 是不同语义。
它必须具备：
- `00-提交状态.json`
  - 唯一真源，只允许在这里修改提交包状态
- `00-提交说明.md`
  - 人类可读的提交包说明，由同步脚本派生
- `01-合并PRD/`
  - 面向人的阅读层；流程图优先、少量文字辅助、可回链到模块真源
  - 默认稳定入口是 `00-PRD索引.md`
  - 启用条件固定为：`已启用完整提交阅读层`
  - 若已启用完整提交阅读层，固定目标文件为 `01-合并PRD阅读层.md`，并由 `00-PRD索引.md` 回链
- `02-原型索引/`
  - 多模块原型入口或索引，由同步脚本派生
- `03-合并评审记录/`
  - 合并后的 AI 深审、评审纪要或阶段性结论，由同步脚本派生
脚本约束：
- 修改 `00-提交状态.json` 后，必须先运行 `scripts/sync_submission_status.py`
- 同步完成后，必须再运行 `scripts/check_submission_consistency.py`
- 若本轮结论还要判断研发是否可直接开工，必须额外运行 `scripts/check_engineering_handoff_readiness.py`
- 若本轮目标是复用最近一次终审证据，必须额外运行 `scripts/check_final_delivery_gate.py`
- Gate B 脚本回写 `engineering_handoff_required / engineering_handoff_ready` 后，必须再补跑一次 `scripts/sync_submission_status.py -> scripts/check_submission_consistency.py`
- final gate 只做 final-review evidence 新鲜度校验，不写 `00-提交状态.json`
- 推送远端前，必须再运行 `scripts/check_publish_boundary.py`
- 上述脚本的唯一维护源位于组织 skill `org_skills/03-submission-status-governance/`；模板与真实项目中的同名文件是发布副本
- `scripts/check_publish_boundary.py` 的唯一维护源也位于同一组织 skill；模板与真实项目中的同名文件是发布副本
- 如果正式 `PRD` 头部也要跟随提交包状态变化，必须在真源的 `artifacts.prd_files` 中登记目标文件，由脚本统一同步，不能人工逐份手改
- 正式 `PRD` 文件名必须状态中立，统一使用 `*-PRD.md`，不得把 `-draft` 写进正式文件名
- 禁止直接手改 `80-完整提交包/` 下的展示层正文，再把它当作事实源
- `80-完整提交包/01-合并PRD/` 如需形成正式阅读层，优先参考组织模板 `assets/templates/36-模板-合并PRD阅读层.md`
- 若已启用完整提交阅读层，必须落到固定文件 `80-完整提交包/01-合并PRD/01-合并PRD阅读层.md`，不得临时散落到其他命名文件
- 合并 `PRD` 阅读层至少应能覆盖：
  - 项目整体业务闭环图
  - 模块关系图
  - C 端主旅程图
  - 后台支撑旅程图
  - 分模块主流程图
- 合并 `PRD` 阅读层默认使用 `Mermaid`；图给人看，结构化文字给 AI 和执行链看
- `scripts/check_engineering_handoff_readiness.py`、`scripts/check_final_delivery_gate.py` 的项目侧说明只保留消费边界，不重述 `32` 层固定验收语法。
## 9. 事实源规则
- `00-项目包/` 是项目事实源。
- `00-项目包/03-系统拓扑与核心名词表.md` 是共享对象、共享状态、共享规则与共享能力宿主。
- `00-项目包/05-项目级原型与高保真标准.md` 是项目级原型表达、保真与画布规则宿主。
- `00-项目包/06-产品级共享资产底座.md` 是共享元件、共享组件与共享 Pattern 宿主。
- `01-模块执行包/` 是需求与模块事实源。
- `4.3 页面合同清单` 是页面合同、页面元素、首屏焦点与页内交互承接的全局第一真源；没有正式 `4.3` 时，不得进入原型链或原型审查链。
- `4.4 结构化契约字段清单` 是结构化业务字段、跨页透传、状态回写与验收字段的全局第一真源；命中时缺少正式 `4.4`，同样不得进入原型链或原型审查链。
- 命中登录、验证码、支付、SLA、删除、第三方接口、动态配置、AIGC 等高风险规则场景时，必须补齐 `4.2A 关键规则参数与边界冻结表`；只有概念、没有具体规则值时，一律判 `未冻结`，按 `Blocker` 回流。
- `80-完整提交包/` 是交付视图。
- `80-完整提交包/00-提交状态.json` 是完整提交包状态的唯一真源。
- `80-完整提交包/01-合并PRD/` 是阅读层，不是模块真源层。
- `80-完整提交包/01-合并PRD/`、`02-原型索引/`、`03-合并评审记录/` 都是派生展示层。
- 每个模块在进入原型链前，都必须先具备 `01-PRD/02-模块流程图与人类确认.md`；原型前总控负责准入，产品审查负责质量兜底。
- 原型前结构核对继续由 `产品结构约束.yaml` 的 `prototype_entry` scope、`产品路径约束.yaml`、`35-过程-总控复核清单.md` 与 `21-审查-产品审查` 共同承接，不新增独立 prototype-entry 运行时脚本门。
- 根 `docs/` 是白名单补充区，不是事实源。
- 历史兼容目录 `90-工具包/` 如仍存在，也永远不是事实源。
- 禁止在 `80-完整提交包/` 中直接修改正文，再回头当作事实源。
## 10. 与项目包 skill 的关系
项目包引导 skill 在 `Phase 4` 完成后，可以继续询问是否派生下游执行目录。
如果用户确认派生，skill 只允许：
- 询问交付模式
- 询问模块名称
- 派生目录骨架
- 派生说明文件
- 同步派生仓库级 `AGENTS.md`
- 同步派生 `docs/00-说明.md` 与 `docs/01-跨模块补充资料/README.md`
- 如果用户选择派生完整提交包，必须同时确保以下文件存在：
  - `80-完整提交包/00-提交状态.json`
  - `80-完整提交包/00-提交说明.md`
  - `80-完整提交包/01-合并PRD/00-PRD索引.md`
  - `80-完整提交包/02-原型索引/00-原型入口.md`
  - `80-完整提交包/03-合并评审记录/00-评审索引.md`
  - `80-完整提交包/03-合并评审记录/30-结果-AI深审记录.md`
  - `scripts/sync_submission_status.py`
  - `scripts/check_submission_consistency.py`
  - `scripts/check_engineering_handoff_readiness.py`
  - `scripts/check_publish_boundary.py`
这些脚本属于组织 skill `03-submission-status-governance` 的发布副本，必须随模板派生进入真实项目，不在真实项目中各自分叉维护。
skill 不允许在这个阶段自动脑补模块 PRD、原型或评审内容。
