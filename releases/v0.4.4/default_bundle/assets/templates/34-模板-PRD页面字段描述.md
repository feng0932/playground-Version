# PRD 页面合同与结构化契约字段模板

| 项目 | 内容 |
|------|------|
| 文档角色 | `33-模板-产品PRD.md` 的 `4.3 页面合同清单` 与 `4.4 结构化契约字段清单` 复用子模板 |
| 文档状态 | `正式条款版 v1` |
| 所属目录 | `install/default_bundle/assets/templates/` |
| 定位说明 | 仅作为 `PRD` 正文中 `4.3` 与 `4.4` 的复用模板，不是独立正式交付物 |
| 更新时间 | `2026-04-04` |

## 1. 使用范围

- 本模板同时用于填写 `PRD` 正文中的 `4.3 页面合同清单` 与 `4.4 结构化契约字段清单`。
- `4.3` 负责冻结页面合同与页面元素，回答“页面为什么存在、首屏必须看到什么、页面上必须有什么”。
- `4.4` 负责冻结结构化契约字段，回答“哪些正式业务字段需要被输入、透传、回写、复用或验收”。
- 页面元素不等于原子字段；展示区、结果区、记录入口、状态提示等对象，应写在 `4.3`，不应硬塞进 `4.4`。
- 只有命中结构化契约字段的页面才进入 `4.4`；纯展示页、纯汇总页、纯提示页不需要为了占位写“无”。
- `4.4` 的字段名称必须使用稳定 canonical key，不允许把展示文案直接当正式字段名。
- 若页面主体展示的是动态业务对象，且该对象参与验收，则 `4.4` 必须冻结对象的最小验收载荷。

## 2. `4.3 页面合同清单` 模板

### 2.1 页面元信息模板

每个页面在写页面合同与页面元素前，先补齐以下元信息：

| 字段 | 填写内容 |
|------|------|
| 页面编号 | |
| 页面名称 | |
| 终端类型 | `手机端` / `PC端` / `管理后台` / `通用` |
| 来源需求编号 | 1. <br>2. |

页面编号必须按页面实例唯一填写；同一业务页面若跨终端出现，必须拆成不同页面编号，不得复用同一编号分多终端行。

### 2.1A 手机端认证页补充冻结项

- 只在手机端认证页命中时填写。
- 若 primary auth method 与 fallback auth method 的用户任务不同，或所需输入不同，必须拆成独立页面。
- 每个手机端认证页都必须补齐：
  - `认证任务`
  - `所需输入`
  - `切换入口目标`
- 不得用抽屉、弹层或同页辅助区替代第二认证页。

### 2.3 页面元素清单

- `页面元素清单` 是 `4.3 页面合同清单` 的正式组成部分，用来冻结页面上必须出现的核心对象。

### 2.4 页面元素类型枚举

| 元素类型 | 说明 | 典型场景 |
|------|------|------|
| `展示区` | 只读展示的页面对象 | 客户摘要区、订单信息区 |
| `输入控件` | 用户主动录入内容的控件 | 文本输入框、多行备注框 |
| `选择控件` | 用户选择模式、标签或筛选条件 | 单选标签、下拉选择、模式切换 |
| `操作入口` | 触发动作的按钮或入口 | 生成按钮、记录入口、新增入口 |
| `结果区` | 承接业务结果或 AI 输出的区域 | 建议结果区、计算结果区 |
| `状态提示` | 承接加载、限制、失败、挂起等状态 | 处理中提示、频控提示、错误提示 |
| `导航入口` | 页面间导航或返回路径 | 返回入口、二级导航、Tab 入口 |

### 2.5 页面元素列说明

| 列名 | 填什么 | 示例 |
|------|--------|------|
| **元素名称** | 用户或设计稿中看到的页面对象名称 | 客户摘要区、记录入口 |
| **元素类型** | 页面对象归属类型 | 展示区、结果区、输入控件 |
| **元素目的** | 一句话说明这个元素服务什么任务 | 展示客户上下文，避免误操作 |
| **是否首屏核心** | 是否属于首屏必须看到的核心元素 | 是 / 否 |
| **出现条件** | 什么时候出现、隐藏或切换 | 生成成功后展示 |
| **是否必须原型承接** | 原型是否必须把它画出来 | 是 / 否 |

## 3. `4.4 结构化契约字段清单` 模板

### 3.1 适用判断

下列任一命中时，页面应进入 `4.4`：

- 用户直接录入或选择该字段
- 字段需要跨页透传
- 字段需要跨模块复用
- 字段需要被记录、回写或回流
- 验收必须逐项核对该字段

### 3.2 页面页头模板

```markdown
### 4.4.X [页面名称] 结构化契约字段

| 字段 | 填写内容 |
|------|------|
| 页面编号 | P-XX |
| 页面名称 | [页面名称] |
| 终端类型 | 手机端 / PC端 / 管理后台 / 通用 |
| 来源需求编号 | 1. REQ-001 <br>2. REQ-002 |

| 字段名称 | 字段语义 | 字段角色 | 生产方 | 消费方 | 必填规则 | 可选值/枚举 | 读写时机 | 持久化/复用方式 |
|------|------|------|------|------|------|------|------|------|
| mode | 当前生成模式 | 输入 | 用户 | AI 沟通生成链路 | 是 | 首次触达/跟进回复/售后安抚 | 提交生成时读取 | 会话内透传 |
| customer_id | 当前绑定客户 ID | 路由 | 上游页面/路由 | 本页、记录页 | 是 | - | 进入页面时读取 | 跨页透传 |
```

### 3.2A 动态业务对象最小验收载荷

若页面主体展示的是动态业务对象，且 QA / FE / BE 需要依赖它完成验收，则除了 route/context key 外，还必须补齐对象最小验收载荷，例如：

- `customer_summary.name`
- `customer_summary.level`
- `conversation_result.copy_status`
- `asset_card.group_id`

这类最小验收载荷仍属于业务契约，不等于 API request / response 或数据库表设计。

### 3.3 字段角色枚举

| 字段角色 | 说明 | 典型场景 |
|------|------|------|
| `输入` | 用户主动输入或选择的正式业务字段 | mode、备注内容 |
| `路由` | 跨页进入本页所依赖的路由或上下文字段 | customer_id、task_id |
| `输出摘要` | 本页产出并需跨模块复用的摘要字段 | result_summary、input_summary |
| `状态` | 需要回写、持久化或验收的状态字段 | copy_status、generate_status |
| `引用` | 当前页面只引用但需要明确契约来源的字段 | template_id、source_record_id |

### 3.4 列说明

| 列名 | 填什么 | 示例 |
|------|--------|------|
| **字段名称** | 系统内稳定使用的正式字段名 | mode、customer_id |
| **字段语义** | 一句话解释该字段表达什么业务含义 | 当前生成模式 |
| **字段角色** | 该字段在业务里的职责 | 输入、路由、状态 |
| **生产方** | 谁产生这个字段 | 用户、路由、后台任务 |
| **消费方** | 谁读取或依赖这个字段 | 本页、下游模块、记录页 |
| **必填规则** | 是否必填，条件必填要写清条件 | 是 / 否 / 是（命中某模式时） |
| **可选值/枚举** | 选择类字段写全量选项；输入类写 `-` | 首次触达/跟进回复 |
| **读写时机** | 何时读取、何时写入、何时回写 | 进入页面时读取，提交时写入 |
| **持久化/复用方式** | 是否跨页、跨模块、持久化 | 会话内透传、记录落库 |

## 4. 补充规则

- `4.3` 必须优先于 `4.4`；没有页面合同，不应直接写孤立字段表。
- 原型、前端和审查链先消费 `4.3` 的页面合同与页面元素，再消费 `4.4` 的结构化契约字段。
- 页面元素是否是“只读展示”与它是否属于结构化契约字段，不是一回事。
- 只要一个对象主要回答“页面上必须有什么”，它应优先留在 `4.3`。
- 只要一个对象主要回答“系统必须如何读写、透传、回写、验收它”，它应进入 `4.4`。
- `4.3 ↔ 6` 必须按 `page_id` 一一对应；`page_id` 与 `terminal_type` 都必须使用正式写法，不得使用非正式兼容占位写法。
- 手机端认证页若存在 distinct auth tasks 或不同所需输入，必须拆成独立页面；每页都要显式写出 `认证任务 / 所需输入 / 切换入口目标`。
- 不得用抽屉、弹层或同页辅助区替代第二认证页。
- 可选值必须写全量选项，不得只写“见原型”或“待补”。
- 条件必填要在“必填规则”列中直接写清条件，不能只写 `是`。
- route/context key 只回答“怎么带过来”；minimum acceptance payload 还要回答“验收时最少要看到什么”。
- payload contract 属于业务契约，不等于 API contract、request-response schema 或存储结构。
