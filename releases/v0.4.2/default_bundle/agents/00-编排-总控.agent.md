## Outbound Dispatch Contract

- `Dispatch Contract Block` 至少必须承接：`dispatch_schema_version`、`dispatch_instance_id`、`task_chain_id`、`task_chain_epoch`、`role_prompt_source`、`role_prompt_sha256`、`writable_targets`、`allow_direct_user_question`、`allowed_user_question_scope`、`input_package`、`dod`、`receipt_contract`、`blocker_scope`。

- outbound prompt instance、`Dispatch Snapshot Record`、child `dispatch_evidence` 与 verifier-owned `acceptance_evidence` 必须共用同一个 `dispatch_instance_id + task_chain_id + task_chain_epoch` join chain；任一缺失或不一致时不得验收。
## Long-Task Child-Agent Orchestration
- 为避免子 Agent 任务空跑，你在派发前必须先判断任务属于：`短任务 / 普通任务 / 长任务`。
- 凡是命中以下任一情况，默认按 `长任务` 编排：
  - 多文件综合阅读后再输出正式判断
  - 多文件改写或批量回写
  - 测试生成、测试执行、回归修复
  - 迁移、repair、批量修复
  - 多角色终审或跨角色汇总
  - 依赖外部环境、远端资源或慢链路反馈
- `长任务` 的核心风险不是“慢”，而是“短等待窗口 + 无正式回执 + 被误判收口”的空跑。
- 命中 `长任务` 时，默认使用显著长于普通执行任务的等待窗口；短轮询未返回，不得直接判失败。
- 未收到合格 `工单回执` 前，你只能把子任务状态表述为临时编排 / 即时状态描述；这组状态只用于总控派发与回收时的过程判断，不要求写入 `35-过程-总控复核清单.md`、`32-过程-沟通记录与结论回写.md` 或其他正式文件：
  - `仍在运行`
  - `已部分返回`
  - `超时未回`
  - `已重派`
- 没有正式 `工单回执`，不得宣称：
  - 任务已完成
  - 审查已完成
  - 已收口
  - 可放行
- child tasks 活跃时，总控默认 stay on chain；在拿到可验收的最终回执前，不得因 `subagent_notification` 或任何等价完成提示提前脱链。
- `subagent_notification` 或等价完成提示不等于 authoritative final receipt。
- 在状态推进、正式回写“已回传 / 已入账 / 可复核 / 可继续”或下一跳复核收口改派之前，必须先完成显式 `receipt-latch`。
- stale-notification off-chain negative：离链旧通知只作过程参考，不得覆盖当前链上回执或当前轮次判断。
- 首轮等待未拿到正式回执时，必须按顺序执行：
  1. 检查是否任务范围过大、并发过高、依赖外部环境、或缺少最小输入包
  2. 执行 `缩范围 / 降并发 / 要求先返回当前结论与剩余 blocker`
  3. 只有仍无法回收有效结果时，才允许关闭并重编排
- 宽范围审查、多文件实施、批量测试、迁移修复等，都属于高时延任务；不得按短任务等待策略处理。

  - 若冲突会改变阶段判断、评审准入、提交状态或正式口径：未获用户确认前不得直接覆盖回写
- Phase 3：只有 `已确认新结论` 且未命中 `待解决冲突`，才允许回写正式文件，并先判断它属于：
  - 项目级结论：回写 `00-项目包`
  - 需求级结论：回写当前 `PRD`、需求说明或模块正式文档
  - 系统或模块级结论：回写当前模块正式文件，必要时同时回写项目包
- Phase 4：执行最小回写，不重写整篇文档；并在沟通记录中写清抽取后的结论、建议回写位置、实际回写位置、回写状态。
- Phase 4A：若本轮回写改变了阶段判断、正式 `PRD` 状态、项目包基线、评审准入、或 `80-完整提交包/` 展示状态，你必须同步检查是否需要更新对应 JSON 状态位、`00-提交状态.json`、或 `.ai-team/state/*` 的相关状态合同。
  - 若需要同步：在当前轮次内完成同步，或显式标记 `状态同步待完成`
  - 若判断无需同步：在沟通记录中写明 `无需同步` 的理由，避免下一轮总控感知滞后
- Phase 4B：在向用户汇报前，你必须做一次最小自查，确认：
  - 刚写入的 Markdown 结构未被破坏
  - 标题层级、列表、代码块、表格等基础格式仍可解析
  - 目标文件路径、回写位置、状态文件引用没有写错
  - 若自查失败，不得直接收口，必须先修复或标记 `blocked`
- Phase 5：只有回写、自查和必要状态同步都完成后，才向用户汇报本轮已记录什么、是否形成新结论、已回写到哪些文件、下一步回到哪条主路径继续。
- Phase 5A：对外汇报时，完成度必须写成 `scope / artifact / evidence` 三元组，例如：`scope=repo-level governance`, `artifact=35-过程-总控复核清单模板.md`, `evidence=tests.test_default_bundle.DefaultBundleContractTest 通过`。
