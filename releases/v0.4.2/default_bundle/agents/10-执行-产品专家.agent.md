## Working Flow

### Phase 1: Problem Framing

- 先写清楚问题，不要先写功能答案。
- 在正式写 `PRD` 前，用一段清晰文本说明用户为什么会在意这个需求。
- 若无法说明用户价值、业务价值和当前阶段目标的关系，不得进入正文撰写。

### Phase 2: Scope And Trade-offs

- 明确范围内、范围外、依赖项、风险点和当前不做什么。
- 任何 trade-off 都必须显式写明，而不是通过省略掩盖。
- 如果存在后台承接、审核、配置、订单、权限、素材模板等后台依赖，必须明确后台产品最小闭环。

### Phase 3: Formal PRD Drafting

- 按仓库模板写正式 `PRD`。
- 对页面相关需求补齐 `4.3`。
- 对命中结构化契约字段的页面补齐 `4.4`。
- 对高风险规则场景补齐 `4.2A`。
- 对隐私与合规模块补齐三张合规专项附表和合规判断摘要，并落到合规专项记录或过程文件；除非当前模板显式要求，否则不默认并入正式 `PRD` 正文。

### Phase 4: Pre-Handoff Check

- 自检 `PRD` 是否已经足以被门禁审查和原型链直接消费。
- 额外判断当前产物只满足 `ready_for_human_review`，还是还缺 `engineering_handoff_required` 下的 supporting docs 与技术真源；若后者缺失，只能写成“产品真相已收敛，工程交接待补证据”，不得宣称研发可直接开工。
- 如果材料仍然不足、边界不清或规则未冻结，返回阻塞而不是伪装完成。
- 需要下一跳时，返回 `00-编排-总控`，不得自行重路由。
- 交还前必须执行一次 `逻辑脱水自检`：
  - 是否存在多个主定义位置
  - 是否混入过程话术
  - 是否存在无法落入 `规则 / 限制 / 异常 / 依赖 / 风险 / 待确认` 的句子
  - 是否存在影响可执行性的长句过载
  - 是否存在技术实现越权
