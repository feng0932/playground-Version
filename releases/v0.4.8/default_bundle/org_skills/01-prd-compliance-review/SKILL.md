# PRD 合规审查

## 概述

在开发前审查产品文档，判断当前 PRD 是否已经具备进入详细设计的合规前提。
重点检查业务范围、账号体系、数据字段、权限能力、内容治理、支付规则和平台约束。
对缺失的规则按风险处理，不把“未写”视为“默认合规”。

## 输入要求

- 执行技能时，优先要求用户提供 `PRD 文档根目录`。
- 审查范围默认是该根目录下的全部文档，而不是只看用户点名的单个文件。
- 如果目录中既有文本文件也有二进制文件：
  - 对可直接读取的文本文件执行全文审查。
  - 对无法直接解析的文件，至少要列入“未解析文档清单”，避免误以为已覆盖全部材料。
- 如果用户没有提供根目录，只给了少量文件，就按当前已提供材料执行，并明确说明这不是“全量目录审查”。

## 工作流

### 1. 确认审查模式

- 默认执行 `PRD 前置合规审查`。
- 如果用户问的是已上线 App、提测包、上架资料、包体行为或市场审核，也要加载更完整的上架审查规则。
- 结论必须和被审查的材料类型一致，不要用 PRD 审查去替代真实上架审查。

### 2. 盘点审查材料

- 先确认用户输入的 `PRD 文档根目录`。
- 递归扫描该目录下的全部文档，不要只看顶层文件。
- 列出扫描到的文件，并区分：
  - 已读取文本文件
  - 暂未读取文件
  - 无法直接解析的文件
- 将材料分类为：
  - 产品概述
  - 用户研究
  - 功能说明
  - 页面说明
  - 业务规则
  - 商业化方案
- 如果扫描结果显示模块不完整，也要把“目录缺失关键文档”作为证据的一部分。
- 明确当前审查范围：
  - `仅 PRD`
  - `PRD + 程序设计`
  - `完整上架审查`

如果当前项目已经有项目包专题入口，还必须优先读取：

- `00-项目包/02-用户画像与权限架构.md`
  - `当前项目账号基线入口`
  - `账号基线能力裁剪表`
  - `前置审查必查项`
- `00-项目包/04-开发前合规附录.md`
  - `当前项目协议与隐私入口`
  - `当前项目实名入口`
  - `当前项目用户治理入口`

若这些项目级专题入口缺失，必须把它们记为项目级前置缺口，不得假设局部模块文档已经等价覆盖。

### 3. 先建证据，再做结论

- 先读标题和章节，判断模块覆盖范围和细化程度。
- 检索以下高风险关键词：
  - `登录` `注册` `账号` `注销` `删除` `撤回同意`
  - `隐私` `个人信息` `权限` `定位` `相机` `相册` `通讯录`
  - `AI` `AIGC` `聊天截图` `客户资料` `上传`
  - `会员` `订阅` `自动续费` `退款` `IAP` `Apple`
  - `举报` `投诉` `未成年人` `青少年` `代理` `激活码`
- 优先使用带文件和行号的证据。
- 审查时区分：
  - `文档已明确`
  - `文档缺失`
  - `文档描述模糊`
  - `文档存在明显合规风险`
  - `需在详细设计阶段继续补齐`

### 4. 选择对应清单

- 审查 PRD 时，始终先加载 [references/prd-precheck-checklist.md](references/prd-precheck-checklist.md)。
- 如果用户要求检查上架规则、应用市场规则或包体行为，再加载 [references/app-review-checklist.md](references/app-review-checklist.md)。
- 如果用户要求完整覆盖 PRD 和上架规则，一并加载 [references/full-checklist.md](references/full-checklist.md)。
- [references/example-prd-review-report.md](references/example-prd-review-report.md) 只用来参考输出结构，不作为事实依据。

### 5. 保守判定风险等级

- `阻断项`：不应进入开发，必须先补齐或修改。
- `高风险项`：建议修改 PRD 后再进入详细设计。
- `提示项`：当前不阻断，但后续大概率引发返工或审查风险。

如果证据缺失，优先判定为 `文档缺失`，不要臆测实现会如何补齐。

### 6. 输出结果

- 先给问题，再给总结。
- 报告必须使用固定章节顺序，避免同一技能在不同任务中输出结构漂移。
- 先给“扫描范围说明”，明确本次是否已覆盖输入根目录下全部可读文档。
- 每条问题至少包含：
  - `编号`
  - `风险等级`
  - `PRD章节`
  - `合规主题`
  - `检查项`
  - `当前问题`
  - `影响`
  - `整改建议`
  - `证据`
- 问题列表后再给：
  - 总体结论
  - 各等级问题数量
  - 优先整改顺序

- 默认将分析结果保存到真实项目的 `04-过程文件/` 中，而不是根 `docs/`。
- 如果当前输入明显属于单一模块目录，默认保存到 `01-模块执行包/<模块名>/04-过程文件/`。
- 如果当前输入属于项目级或跨模块材料，默认保存到 `00-项目包/04-过程文件/`。
- 默认文件名使用清晰的项目化命名，例如：`<项目名>-prd-precheck-report.md`。
- 如果用户显式指定了保存路径或文件名，以用户要求为准。

## 关键判定规则

### 账号体系

- 只要 PRD 支持创建账号，就要检查是否前置考虑了账号注销。
- 如果不同文档对登录方式的描述不一致，要直接报出不一致问题。
- `Sign in with Apple` 是条件触发，不是所有 iOS App 一律必须接入：
  - 如果 iOS 方案包含 `微信 / QQ / Google / Facebook / X / LinkedIn / Amazon` 等第三方或社交登录，并将其作为主账号登录方式，就要要求评估并通常预留 `Sign in with Apple`。
  - 如果 PRD 只使用自有账号体系或手机号登录，不要把 `Sign in with Apple` 判成必需。

### 数据与权限

- 如果文档没有明确写清数据字段、用途、必要性和拒绝后处理，就不要默认视为合规。
- 如果产品涉及粘贴客户消息、上传聊天截图、客户资料沉淀、AI 人设照片分析，要在放行前要求数据边界规则。
- 如果 PRD 使用定位、照片上传、相机、麦克风、通讯录或其他高敏能力，要明确业务触发点和拒绝后的替代方案。

### 内容与 AI

- 如果产品会生成回复建议、主动开场、社交内容、图片或视频，要检查输出边界和治理规则。
- 如果业务场景本身风险较高，要要求补齐禁止生成内容、投诉举报、审核或降级兜底规则。

### 支付与商业化

- 如果商业模式包含会员、订阅、激活码、代理或分销，要在进入设计前明确平台支付约束。
- 如果 iOS 数字商品在范围内，要在 PRD 阶段明确考虑 Apple 支付规则，而不是留到开发后处理。
