| 更新时间 | `2026-03-26` |

## 1. 使用定位

- 本清单不是让产品经理代替技术角色做强判断。
- 本清单的作用是：在上会前，把关键工程与测试风险点提前发给相关角色，收集反馈，再回写到正式记录里。
- 本清单不是最终结论文档；角色反馈必须回写到 `问题响应记录`、`业务取舍说明`、`PRD` 正文、原型或其他正式文档。
- 本清单建议先由 `24-审查-跨角色预审.agent.md` 基于现有材料生成初稿，再由产品经理校对后发送。
- 若命中 `异步任务 / 删除保留周期 / 动态权限 / 动态配置 / 发布下线 / 隐私请求 / 跨页恢复 / 多分支主链 / 状态数>=4` 任一触发器，本清单默认从“建议使用”升级为“必须生成”。

## 2. 使用规则

1. 产品经理负责发出问题，不负责替前端、后端、测试、DBA、运维拍板。
2. 建议由 AI 先生成本清单初稿，产品经理负责校对、删减、补充接收人与时间点。
3. 每个问题必须有明确应答角色、回复时间点、回写位置。
4. 若问题涉及多角色，必须写清主责回答方。
5. 若当前团队没有对应角色，必须明确临时代答人，并记录“临时代答”。
6. 阻塞级问题未得到明确回应前，不建议进入正式评审会。
7. 本清单也是 Gate B 的 unresolved issue ledger 候选载体；owner、最晚反馈时间、回写位置缺一不可。
