# 中国市场 App PRD 前置合规检查清单

## 目标

本清单用于在 `正式开发前` 检查 `PRD` 是否已经埋下中国市场 App 上架与监管合规风险。

这里检查的是：

1. `PRD 是否写清楚必要的合规前提`
2. `PRD 是否存在明显会导致后续无法合规落地的设计`
3. `PRD 是否遗漏关键合规流程、角色、数据、权限、内容治理或商业化约束`

这里暂不检查：

1. 代码是否已实现
2. 隐私政策文案是否已发布
3. SDK 是否已真实接入
4. 上架截图、测试账号、软著、许可证是否已提交
5. 提测包是否通过实际巡检

## 适用范围

适用平台：

1. `iOS / App Store`
2. `HarmonyOS / 华为应用市场`
3. `国内主流 Android 市场`

适用输入材料：

1. `PRD`
2. 功能列表
3. 页面清单
4. 用户流程图
5. 原型说明
6. 商业化方案
7. 业务规则说明

## 建议输出字段

建议技能在检查 PRD 时，对每个问题输出以下字段：

- `编号`
- `风险等级`
- `PRD章节`
- `合规主题`
- `检查项`
- `PRD应体现内容`
- `当前问题`
- `影响`
- `整改建议`
- `适用平台`

风险等级建议：

- `阻断项`：不补齐就不建议进入开发
- `高风险项`：建议修改 PRD 后再进入详细设计
- `提示项`：建议补充，否则容易在设计、开发、提审阶段返工

## 判定原则

PRD 阶段的检查重点不是“是否已经合规上线”，而是判断：

1. `是否有关键合规信息缺失`
2. `是否有明显违规的产品设定`
3. `是否缺少后续必须落地的合规流程`
4. `是否遗漏平台硬约束，导致后续必然返工`

技能在输出结论时，应区分：

- `文档已明确`
- `文档缺失`
- `文档描述模糊`
- `文档存在明显合规风险`
- `需在详细设计阶段继续补齐`

## 直接判定为不建议进入开发的情况

满足以下任一情况时，技能应输出 `不建议进入开发`：

1. PRD 无法判断业务类型，无法套用监管口径
2. PRD 涉及受监管行业，但完全未提许可、备案或准入约束
3. PRD 设计要求收集明显超出业务必要性的个人信息
4. PRD 将高敏权限设计为首次启动默认申请
5. PRD 包含账号体系，但完全未考虑注销、删除、投诉、撤回路径
6. PRD 涉及 UGC、社交、直播、AIGC，但没有任何内容治理机制
7. PRD 的商业化方案明显违反平台规则或存在违法经营风险
8. PRD 明确依赖违规后台保活、静默下载、规避分发等技术路径

## 判定为建议补充 PRD 后再进入设计的情况

满足以下任一情况时，技能应输出 `建议补充 PRD 后再进入设计`：

1. 用户角色、年龄层、游客与登录边界不清
2. 数据项未拆到功能级，无法判断最小必要
3. 权限场景未明确，无法判断何时申请、拒绝后如何处理
4. 支付、订阅、会员权益、退款、恢复购买规则不完整
5. 投诉、举报、客服、注销、删除等路径未预留
6. 个性化推荐、AI 生成、内容分发逻辑过于模糊
7. 平台专项约束未体现，未来大概率返工
