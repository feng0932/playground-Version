# PRD 需求理解分析框架

## 1. 文档定位

这份参考文件用于说明 PRD 需求理解阶段为什么要做、重点要看什么，以及分析结果应该如何组织。

它是分析框架，不记录某个具体项目的事实细节，也不替代后续的流程文档、问题清单或技术设计文档。

它主要用于技术设计前的需求澄清阶段，帮助团队先判断需求是否已经足够清晰、闭环、可讨论，而不是直接进入方案设计、任务拆解或排期评估。

## 2. 需要回答的核心问题

PRD 需求理解阶段，本质上要回答 3 个问题：

1. 这个需求的流程是否闭环，是否存在需求孤岛、流程断点或异常后无处回退的情况。
2. 这个需求描述是否完整、合理、前后一致，是否存在遗漏、冲突、边界不清或规则无法执行的问题。
3. 这个需求里的业务领域是否划分清楚，领域内和跨领域的字段、概念、状态是否存在重复定义、语义相似或边界混用的问题。

把问题收束成这 3 类，有两个目的：

- 避免需求理解阶段被拆成一长串松散检查项。
- 让后续输出文档都能回到统一框架，而不是各写各的。

## 3. 三类核心检查框架

### 3.1 流程闭环检查

流程闭环检查关注的是需求链路是否完整可走通，而不是单纯把流程画出来。它需要回答：用户如何进入流程、如何完成目标、遇到异常后如何处理、失败后如何回退、状态如何继续流转。

检查时至少覆盖以下 3 类路径：

- 主流程：需求在正常条件下如何完成。
- 异常流程：输入异常、权限不足、状态异常、依赖失败时如何处理。
- 回退流程：用户取消、失败重试、中途退出后如何恢复或回到上一步。

这一类检查重点识别的问题包括：

- 某个功能点被单独提到，但没有前置入口或后续承接，形成需求孤岛。
- 流程能开始但不能完成，或能失败但没有恢复路径，形成流程断点。
- 状态变化被提到，但缺少触发条件、回退条件或后续影响说明。

### 3.2 需求完整性与一致性检查

这一类检查关注需求文本本身是否足够清晰、可靠、可执行，可以拆成 3 个层面：

- 完整性：目标、范围、角色、规则、状态、异常条件是否齐全，是否有遗漏。
- 合理性：业务规则是否成立，约束是否现实，描述是否具备执行条件。
- 一致性：不同章节、不同页面、不同角色视角下的描述是否前后一致，是否存在矛盾。

这一类检查重点识别的问题包括：

- 关键前置条件、限制条件、异常条件缺失。
- 同一需求在不同章节出现规则冲突、状态矛盾或角色职责冲突。
- 术语漂移，同一个概念在不同地方被不同名称表达，或不同概念被同一名称复用。
- 范围边界不清，导致“做什么”和“不做什么”无法稳定判断。

### 3.3 领域划分与语义相似性检查

这一类检查的前提是先划分业务领域，再分析字段、概念、状态和规则。没有领域边界，很多“看起来差不多”的定义其实无法判断是复用、重复，还是冲突。

检查时需要区分两种情况：

- 同一领域内：重点检查同义不同名、重复定义、状态含义重叠。
- 不同领域之间：重点检查名称相似但语义不同、字段结构相似但业务边界不同、概念被错误复用。

这一类检查重点识别的问题包括：

- 同一领域内存在含义相近但命名不同的字段、概念或状态。
- 不同领域之间存在看似相似、实际语义不同的字段或状态。
- 术语、字段、状态被跨领域错误继承，导致边界混用。
- 某些定义在一个领域成立，但被直接套用到另一个领域后失真。

## 4. 分析结果应如何下沉

这份框架只定义分析方法，不承载具体项目事实。实际分析时，结果继续下沉到不同类型的输出文档中：

- 流程闭环检查：主要落到流程、状态流转、异常回退类文档。
- 需求完整性与一致性检查：主要落到待确认问题、冲突项、缺失项、一致性检查类文档。
- 领域划分与语义相似性检查：主要落到术语、字段、状态、领域边界分析类文档。

如果存在单独的范围文档，那么它回答的是“这次具体分析哪些材料、覆盖哪些范围”；而本框架回答的是“需求理解阶段通常应该重点检查哪些问题”。

## 5. 适用边界与非目标

这份框架适合作为跨项目复用的需求理解总纲，用于统一团队在技术设计前的分析视角。

它不解决以下问题：

- 不替代具体项目的分析范围说明。
- 不直接输出技术方案、接口设计或数据库设计。
- 不直接拆解研发任务、测试任务或排期计划。
- 不替代页面级程序设计说明，也不展开到实现细节层。
