## 1. 模板用途

- 本模板用于记录 Gate B 的 trigger-based `IF-THEN` 门槛，不替代职责矩阵。
- 本模板只在 `ready_for_human_review=true` 后使用。
- 本模板明确区分 `required supporting docs` 与 `optional supporting docs`。
- 本模板要求 file-system existence check 与 semantic minimum check 同时成立。
- 本模板要求显式记录 legacy migration record，避免把旧模块自动抬成 `engineering_handoff_ready=true`。
- 所有对人可见的工程交接正文，默认使用自然、流畅的简体中文；英文只保留给稳定 machine identifier。
- 若对外入口中的 `02-工程交接/README.md` 仍保留 starter 口径，应直接判为 `starter-handoff contradiction`，不得放行。

## 2. 使用规则

1. Gate B 是 boolean gate，不是新的 PRD 生命周期阶段。
2. trigger 以产品真相推导为准，作者自述只能辅助，不能压制已推导出的 trigger。
3. 一个 trigger 命中后，必须追加对应 required supporting docs，不得被 reading-layer 摘要替代。
4. 每个 required supporting doc 都要写明 `owner_role / path / version / linked_prd_source`。
5. 任何 `N/A` 都必须符合 registry 允许范围，并写出简短理由。
6. 任何文件存在但语义不足的情况，Gate B 一律按缺失处理。
7. `PM` 给人看的模块图必须首先服务于人类协作理解，不能被 `BE` 状态真源替代。

## 5. Trigger 门槛

| IF | THEN required outputs | 备注 |
|------|------|------|
| `async_task=true` | `pm_module_diagram + state_flow_diagram + timeout_policy + retry_or_compensation_policy` | 先用 `PM` 模块图统一主链、异步等待与恢复出口，再补状态、超时、重试或补偿 |
| `delete_or_retention=true` | `lifecycle_matrix + retention_rule + cleanup_owner` | 需要写清 delete / archive / restore 边界 |
| `dynamic_permission=true` | `role_matrix + data_scope_rule + audit_rule` | 需要写清角色边界与审计规则 |
| `dynamic_config=true` | `effective_rule + inflight_policy + rollback_rule` | 需要写清变更生效与回滚 |
| `publish_or_offline=true` | `pm_module_diagram + publish_state_flow + block_type_mapping + repair_page_mapping` | `PM` 先补给人看的模块图，`Ops/SRE` 再补发布流与修复入口 |
| `cross_page_recovery=true` | `pm_module_diagram + recovery_flow_or_state_diagram` | `PM` 图先说明恢复入口、回跳与结果出口，`FE/BE` 再补技术恢复流 |
| `privacy_request_or_complaint=true` | `pm_module_diagram + request_state_flow + sla_rule + pause_resume_rule + result_feedback_rule` | 先统一人类理解，再补状态与 SLA 真源 |
| `branching_mainline_ge_3=true` | `pm_module_diagram` | 需要一张给人看的模块图，说明主链、分叉与汇合 |
| `formal_status_count_ge_4=true` | `pm_module_diagram + status_transition_diagram` | 状态数达到四态或以上时，既要有给人看的模块图，也要写清状态列表与禁止迁移 |

- `required supporting docs` 必须全部落表。
- `optional supporting docs` 可以补充，但缺失不应单独构成 Gate B 阻塞。
- 若路径、类型、owner 或 `linked_prd_source` 失配，应按 registry mismatch 处理。

## 7. 迁移与阅读层

- `legacy migration record` 只记录迁移事实，不自动把旧模块判成 `engineering_handoff_ready=true`。
- `阅读层` 只能帮助人理解，不得替代 trigger-required module artifact。
- 若阅读层里画了图，但 `PM` 给人看的模块图缺失，Gate B 仍然失败。
- `PM` 给人看的模块图不能被 `BE` 状态真源替代；二者职责不同。
- 若 `linked_prd_source` 指向 `80-完整提交包/` 展示层，应判 `display-layer linked-truth conflict`。
- 若历史真源缺少明确 `retired-truth marker` 或替换关系，应判 `retired-truth ambiguity`。
- 任何 `N/A` 只在 registry 与 scope 同时允许时才有效。

## 8. 收口要求

- `engineering_handoff_ready=true` 之前，必须先确认所有 required supporting docs 已通过 file-system check 与 semantic minimum check。
- 发现 owner 不明、路径不符、版本缺失、回链缺失或 trigger 漏判时，必须写入 `gate_b_blockers`。
- 若本模板用于审查输出，结论必须可回链到 `artifact_owner_map` 与 `required_supporting_docs`。
