## 提交状态机制

- `80-完整提交包/00-提交状态.json` 是当前提交包状态的唯一真源。
- `01-合并PRD/00-PRD索引.md`、`01-合并PRD/01-合并PRD阅读层.md`、`02-原型索引/00-原型入口.md`、`03-合并评审记录/00-评审索引.md` 只展示由真源派生出的状态，不应再手工分别维护。
- 如需同步正式 `PRD` 头部状态，可在真源的 `artifacts.prd_files` 中登记目标文件，再运行同步脚本。
- `scripts/sync_submission_status.py`、`scripts/check_submission_consistency.py`、`scripts/check_engineering_handoff_readiness.py` 来自组织 skill `org_skills/03-submission-status-governance/` 的发布副本。
- 同步命令：`python3 scripts/sync_submission_status.py`
- 校验命令：`python3 scripts/check_submission_consistency.py`
- Gate B 命令：`python3 scripts/check_engineering_handoff_readiness.py --input <gate-b-json> --project-root <项目根>`
- final gate 命令：`python3 scripts/check_final_delivery_gate.py --project-root <项目根>`
- Gate B 写回：`check_engineering_handoff_readiness.py` 只允许回写 `engineering_handoff_required / engineering_handoff_ready` 到 `00-提交状态.json`，不得改写 `phase`
- 标准顺序固定为：先改 `00-提交状态.json`，再跑 `sync`，最后跑 `check`。
- 若当前目标是工程交接判断，再运行 Gate B 校验，并在 Gate B 写回后补跑一次 `sync -> check`；`ready_for_human_review` 不等于 `engineering_handoff_ready`。
- 若当前目标是复用最近一次终审证据，再运行 final gate；它只做新鲜度校验，不写 `00-提交状态.json`，也不独立授予“可最终提交”。
