## 1. 模板用途

- 本模板只覆盖第一版最小实操闭环：`PRD 起草 -> PRD 评审`。
- 本模板的目标是让人直接开始写、直接提交、直接对照门禁，不需要先理解全部规则树。
- 本模板不覆盖 `PRD 冻结` 的完整填写；冻结只在门禁说明中作为下一步提示。

## 2. 使用顺序

1. 先看当前 bundle 的总控入口与流转说明，按统一路径开始执行。
2. 再打开本模板，按模板填写 `draft` 包。
3. 填写前，先回读本项目仓库中的 `00-项目包`。
4. `PRD` 完成后，`6. 交互原型` 章节必须保留；如需原型，再按终端类型选择对应原型提示词；若本轮不涉及原型，显式写 `无，本轮不涉及原型`。
5. 再进入当前 bundle 的门禁与审查路径，按说明执行硬门禁。
6. 若本轮目标是提交 `PRD 评审`，再配合当前 bundle 内的审查 Agent 与结果模板完成闭环。
7. 通过后，整理成评审版 `PRD` 与评审资料，进入 `PRD 评审`。

## 2A. 正文写作压缩规则

- 正式 `PRD` 默认使用“规则体”写法，不使用“解释体”“说服体”“感悟体”。
- 每章只回答一个问题；同一业务规则只能有一个主定义位置。
- `4.2 / 4.2A / 4.3 / 4.4 / 7` 承担正式规则定义；`5 / 9 / 10` 不得复写正式规则正文。
- 模板说明、评审说明、过程说明、写作说明不得进入正式 `PRD` 正文。
- 事实性背景、明确取舍、可量化依赖说明可以保留，但必须直接服务决策、范围或验收。
- 发现长句同时塞入背景、规则、异常和依赖时，应拆成短规则句，不保留说明文写法。

##### 4.3 页面合同清单

- `4.3 页面合同清单` 是 `PRD` 正文中的正式章节，不是可选候选草案。
- 凡是需求涉及页面、原型、页面结构、页面元素、交互承接或状态承接的页面，都必须补齐页面合同清单。
- 页面合同前置材料模板：
  - 主任务
  - 首屏必看
  - 不能抢焦点内容
  - 必须页内完成的交互
  - 统一词库
- 页面合同前置材料必须围绕当前页面自身，不要复写整套项目愿景或接口设计。
- `4.3` 负责冻结页面合同与页面元素，不负责替代完整数据模型、接口字段或技术实现细节。
- 每个页面的合同清单必须先写页面元信息，再写页面合同和页面元素清单；页面元信息至少包含：
  - 页面编号
  - 页面名称
  - 终端类型：`手机端` / `PC端` / `管理后台` / `通用`
  - 来源需求编号：至少关联 `1` 个 `4.2` 中的需求编号
- 页面元素清单用于告诉设计、前端和审查链“页面上必须有什么”，不是把所有页面元素都伪装成原子字段。
- 页面元素类型统一使用：`展示区` / `输入控件` / `选择控件` / `操作入口` / `结果区` / `状态提示` / `导航入口`
- 页面合同与页面元素的复用子模板、元素类型枚举、列说明与示例，统一参考 [34-模板-PRD页面字段描述.md](34-模板-PRD页面字段描述.md)。
- 页面编号必须是该页面在当前终端下的唯一编号；同一逻辑页面若跨终端出现，必须拆成不同页面编号。
- `4.3 ↔ 6` 必须按 `page_id` 一一对应；`page_id` 不得跨页复用，`terminal_type` 只能取正式终端类型，不得使用非正式兼容占位写法。
- 若页面命中 `跨页恢复 / 多分支主链 / 状态数 >= 4 / 异步任务 / 发布或下线 / 权利请求` 任一条件，必须补充单独的 `PM` 给人看的模块图；图首先给人看，用于统一业务主链、关键分叉、恢复出口和角色交接理解，阅读层图与 `BE` 状态真源都不能替代它。
- 禁止把阻断、异常、回跳、兜底类结果节点写成抽象口号；若图或正文里出现 `限制进入核心能力 / 回来源页 / 异常处理 / 失败兜底`，至少要同时写清用户可见承接页或承接状态，以及允许出口或可执行下一步。
- 若页面属于手机端认证页，且 primary auth method 与 fallback auth method 的用户任务或所需输入不同，必须拆成独立页面；每页都必须冻结 `认证任务`、`所需输入`、`切换入口目标`，且不得用抽屉、弹层或同页辅助区替代第二认证页。

##### 4.4 结构化契约字段清单（命中时必填）

- `4.4 结构化契约字段清单` 是 `PRD` 正文中的正式章节，但只在命中结构化契约字段时填写。
- `4.4` 只记录直接影响录入、选择、跨页透传、跨模块复用、状态回写或验收判定的正式业务字段。

- 结构化契约字段的复用子模板、字段角色枚举、列说明与示例，统一参考 [34-模板-PRD页面字段描述.md](34-模板-PRD页面字段描述.md)。
- 页面编号必须与 `4.3 页面合同清单` 保持一致；若同一逻辑页面跨终端出现，必须拆成不同页面编号。

- `required supporting docs` 只列业务上确实需要的正式下游材料，不把候选草案混进去。

- `canonical backlink` 必须回到当前模块 `PRD` 真源，不得把阅读层、展示层或汇报层反向写成真源。
- `linked_prd_source` 只允许指向模块级 `PRD` 真源，不得指向 `80-完整提交包/` 展示层。
- 模块进入原型链前，`required supporting docs` 必须登记 `01-模块执行包/<模块名>/01-PRD/02-模块流程图与人类确认.md`。

- 若旧真源继续保留，必须显式写 `retired-truth marker`，说明当前 canonical、历史保留和替换关系；两个文件都写 `approved` 但没有退役标记，直接视为歧义。

## 5. 填写要求

- `PRD 正文` 按中文字段名填写；英文结构 key 不在模板中显式出现，正式结构校验与字段归位以 `产品结构约束.yaml` 和 `34-模板-PRD页面字段描述.md` 为准。

- 每章只回答一个问题；同一规则只允许一个主定义位置，后续章节只允许引用或摘要。
- `4.3 页面合同清单` 与 `4.4 结构化契约字段清单` 都是正式结构，不得降级成聊天说明、口头补充或候选草案。
