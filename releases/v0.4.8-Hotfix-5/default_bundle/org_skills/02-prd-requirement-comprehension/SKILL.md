# PRD需求理解分析

## 概述

不要把这项工作做成摘要整理或功能罗列。先界定分析范围，再按统一框架判断需求是否闭环、是否完整一致、以及业务领域边界是否清楚。

所有“缺失、冲突、语义重叠、边界混用”判断都必须附证据。输出时保持章节、表格列和结论顺序稳定，避免每个项目写成不同结构。

## 执行流程

1. 读取需求材料，列出输入文档和覆盖边界。
2. 读取 `references/analysis-framework.md`，按三类核心检查组织分析。
3. 读取 `references/output-template.md`，按固定章节和表格结构输出。
4. 读取 `references/analysis-rules.md`，应用取证规则、判断规则和写作规则。
5. 先识别哪些内容只有模块级说明，哪些内容已经进入功能级、页面级或状态级规则，避免把“写得多”误判成“已闭环”。
6. 输出结论时，同时判断：
   - 全量产品 / 全量版本是否具备进入技术设计阶段条件
   - 某些局部模块是否已接近可设计状态

## 强制要求

- 使用中文输出。
- 明确区分“源文档事实”“基于事实的分析判断”“需要确认项”。
- 对每条关键缺口、冲突、语义重叠或边界混用，给出直接证据。
- 证据格式统一为 `文档名:行号`；如果暂时无法定位行号，至少给出明确章节。
- 对“冲突”至少引用两处直接证据。
- 对“缺失”要同时说明“期望来自哪里”和“当前缺在哪里”。
- 分析结果默认保存到真实项目对应的 `04-过程文件/`；如果用户明确指定了输出路径，则以用户要求为准。
- 不输出技术方案、接口设计、数据库设计或任务拆解。
- 不引用技能外部文档作为框架依据；仅使用本技能内部参考文件。

## 资源

- `references/analysis-framework.md`
  用于定义分析阶段要检查什么。
- `references/output-template.md`
  用于固定输出章节、表格列和结论顺序。
- `references/analysis-rules.md`
  用于约束取证方法、判断边界和标准写法。

## 结果标准

- 输出文档的章节和细节粒度要稳定一致，不因项目不同而随意改结构。
- 如果某个章节暂时无法填实，不要删章节；写清“当前材料不足以支持判断”。
- 如果材料只覆盖局部模块，不要把局部完备误写成全量版本完备。
