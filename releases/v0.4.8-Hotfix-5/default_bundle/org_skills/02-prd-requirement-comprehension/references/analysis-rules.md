# 分析与写作规则

## 1. 范围判断规则

- 先列出输入材料，明确本次分析对象包含哪些文档。
- 先判断每份材料处于什么粒度：
  - 项目背景 / 用户画像
  - 产品架构 / 模块清单
  - 功能规则
  - 页面说明
  - 后台说明
- 不要把“模块级说明很多”误判成“需求已经闭环”。
- 如果只有局部模块写得细，明确写出“全量产品不完整，但局部模块接近可设计”。

## 2. 取证规则

- 对关键判断必须给证据。
- 证据格式统一为 `文档名:行号`。
- 如果暂时无法获取行号，至少写到明确章节；拿得到行号时优先写行号。
- 对“冲突”至少提供两处直接证据。
- 对“缺失”同时提供两类信息：
  - 期望来自哪里
  - 当前缺在哪里
- 对“语义重叠”至少给出两种命名或两处定义。
- 对“边界混用”同时给出：
  - 原本的领域或模块归属证据
  - 当前混用表现的证据

## 3. 判断规则

### 3.1 什么时候写“缺失”

满足以下任一情况可判为缺失：

- 用户旅程或模块目标已经提出，但没有对应页面级 / 功能级承接。
- 页面存在主流程，但没有异常、回退或恢复路径。
- 后台能力已定义，但前台行为没有对应规则。
- 状态被提到，但缺少触发条件、回退条件或后续影响。

### 3.2 什么时候写“冲突”

满足以下情况才写“冲突”：

- 同一对象在不同文档中被赋予互不兼容的规则。
- 同一流程在不同文档中给出不同入口、不同出口或不同前置条件。
- 同一页面或模块在不同文档中归属不一致。

不要把“一个写得粗、一个写得细”直接写成冲突。那通常是粒度不一致，不一定是矛盾。

### 3.3 什么时候写“语义重叠”

满足以下情况可判为语义重叠：

- 同一业务对象被多个名称描述。
- 不同层级概念共用同一术语。
- 多个字段都在表达类似含义，但边界没有说明清楚。

### 3.4 什么时候写“边界混用”

满足以下情况可判为边界混用：

- 一个对象在文档 A 属于领域 X，在文档 B 又被当作领域 Y 的组成部分。
- 平台聚合层承担了业务生产层才应该承担的定义。
- 支撑层文档中混入业务层或平台层内容。

## 4. 写作规则

- 使用中文。
- 先写分析范围，再写总体判断，再进入三类核心检查。
- 保持章节、表格列和结论顺序稳定，不要每次分析都重组结构。
- 明确区分：
  - 源文档事实
  - 基于事实的判断
  - 需要确认的问题
- 不把推断写成已经被产品确认的事实。
- 结论段必须区分：
  - 全量产品 / 全量版本是否具备进入技术设计阶段条件
  - 某些局部模块是否具备进入技术设计阶段条件

## 5. 完成前检查清单

在交付分析文档前，至少确认以下事项：

1. 是否已经覆盖三类核心检查。
2. 是否已经写清分析范围和材料覆盖边界。
3. 是否所有关键“缺失 / 冲突 / 语义重叠 / 边界混用”都附了证据。
4. 是否避免把技术方案或实现设计混入需求理解分析。
5. 是否明确区分了“全量产品判断”和“局部模块判断”。
