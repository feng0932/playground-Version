| 更新时间 | `2026-04-06` |

## 1. 模板用途

- 本模板用于冻结 Gate A / Gate B 下的 artifact-level 责任边界，不替代门槛规则本身。
- 本模板只回答“谁对哪个真源负责”，不回答“门槛是否通过”。
- 本模板明确区分 `ready_for_human_review` 与 `engineering_handoff_ready`，两者不是同一个含义。
- 本模板适合在 `PRD`、预审讨论、门禁复核与工程交接准备阶段共同引用。
- 所有对人可见的工程交接正文，默认使用自然、流畅的简体中文；英文只保留给稳定 machine identifier。
- `02-工程交接/README.md` 一旦目录内已有真实材料，就必须升级为正式目录索引，不再允许保留“未完成工程交接”的 starter 提示。

## 2. 使用规则

1. 职责按 artifact 冻结，不按阶段口径笼统分摊。
2. 每个 artifact 只能有一个明确 `A`；多人参与时仍需写明唯一 `A`。
3. `PM` 只对产品真相 artifact 取 `R/A`，不对技术真相 artifact 充当默认 `A`。
4. `产品真相` 与 `实现真相` 必须拆开维护，不能混写成同一责任边界。
5. `FE / BE / QA / DBA / Ops/SRE` 只对各自技术真源取 `R/A`。
6. `总控` 在 Gate B 中是 `G = gate executor`，不是技术真源的代签 approver。
7. trigger-required `PM` 给人看的模块图，仍然属于产品真相 artifact；阅读层只能辅助理解，不能替代主 artifact。
8. `PM` 给人看的模块图，用于帮助人统一理解业务主链、关键分叉、恢复出口和角色交接，不能被 `BE` 状态真源替代。

## 3. 角色符号

| 符号 | 含义 |
|------|------|
| `R` | 负责产出或维护该 artifact |
| `A` | 对该 artifact 的最终正确性负责 |
| `C` | 在冻结或通过前参与咨询 |
| `I` | 需要被告知状态或结果 |
| `G` | gate executor；执行门禁判断，但不是 artifact approver |
| `R/A/C/I/G` | artifact 级责任组合写法，不是角色口号 |

## 4. Gate 语义

- Gate A runtime 名称固定为 `ready_for_human_review`。
- Gate B runtime 名称固定为 `engineering_handoff_ready`。
- Gate B 不是新增生命周期阶段。
- `ready_for_human_review=true` 不等于 `engineering_handoff_ready=true`。
- 若 artifact 责任不清，Gate B 应保持阻塞，而不是默认放行。

## 6. 填写要求

- 若某个 artifact 没有清晰 `R/A`，先阻塞，再补责任，不要用“大家共担”代替唯一责任人。
- `PM` 不承担 `API / request-response / data model / DDL / test script` 的真源责任。
- `PM` 给人看的模块图首先是给人看的，不是为了给 validator 凑 token。
- `PM` 给人看的模块图不能被 `BE` 状态真源替代；两者都存在时，也不得合并成同一份 artifact。
- `阅读层`、`说明层`、`摘要层` 不能取代模块级真源 artifact。
- 项目级与模块级 `02-工程交接/README.md` 都属于一等对外入口，必须能一眼指出当前有效真源与历史保留文件。
- 若本轮只需要说明 Gate A / Gate B 的边界，也应保持 artifact 级别写法，不要退回成角色印象分配。
